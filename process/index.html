# Process

This page will provide a brief overview of the general process for our projects.

If you don't see a topic that you'd like more information on, feel free to let the team know and we'll create tutorials/guides on the subject. However, some topics mentioned here will be broken down onto other pages if they are too expansive to cover on a singular page.

So make sure to check the sidebar or use the search function!

# Exploration

If you're reading this as a starting developer, you'll be less involved in this step. It'll usually be the director of operations, project manager, and lead developer who will participate in client meetings to discuss and determine the requirements and scope of the project.

# Design

At some point after client interviews, the design team will start prototyping wireframes (rough drafts for UX/UI layouts). After a few cycles of review, they'll move forward with creating higher fidelity mockups. As design proceeds to the mockup phase, developers will start building in tandem with the design team in an iterative fashion.

Try to keep HTML/CSS work minimal or restricted to basic layout. We don't want to lose too much work/time if there are design changes.

# Development

Now the list I'm providing below is the process for a typical project. You'll skip certain steps or completely omit them depending on the client requirements. However, if we're building a new site from the ground up, you'll usually use a majority these sections.

# Site Setup

Depending on the requirements (determined during the exploration phase), we'll have decided on the technology stack we'll use.

A typical project will use either Craft or WordPress as a content management system. There will be breakdown tutorials on how to set up both.

If there's a pre-existing site, there might not be a need to set up a production server. However, there might still be a need to set up a staging environment. You'll also have to investigate how to connect and push code onto those existing systems. If you're unsure, feel free to double check with the project manager or lead developer.

# Data Migration

Building on the previous point -- if there's a pre-existing site, there's a high likelihood of importing data over to the new site. We'll have some guides on how to do this for Craft and WordPress.

# Back-End Development (Data/Field Building)

After server setup (or pulling in existing architecture), this is where we start defining the data required for what we're building. Craft CMS has docs on how to do this here. In WordPress, we typically use a plugin called Advanced Custom Fields. The great thing about about content management systems is that there's rarely a need to interact with databases directly -- you won't be creating tables and defining columns manually. So if you don't know much about SQL, you're in luck!

(On a serious note, you should brush up on your SQL either way. It's great for building understanding on how all this stuff works under the hood.)

# Front-End Development

Once designs become finalized, you can begin doing more in-depth work for the HTML/CSS/Javascript portion of the site.

I'd like to point out is it can be difficult for mockups to capture movement in UX/UI. You'll want to pull a designer aside for a quick meeting -- find out what elements have animations, sticky positions, fixed positions etc. Feel free to discuss your own cool ideas as well!

There's abundant amounts of resources available everywhere for HTML/CSS/Javascript, all quite extensive (so I'm not going to link any). It'll probably be more beneficial instead to start learning template systems for Craft CMS and WordPress.

(P.S. Feel free to always pull another developer aside during front-end development. Many front-end solutions are open-ended, so it can be hard to decide on a single solution.)

# Testing

You should always be user testing as you work (after all, how will you know what you're building is working?). I (Tim) repeatedly flip between my code and the browser to check if the code is functioning as intended.

One tip -- make sure you're not always following the steps that you intended the user to follow. Do things that a typical user wouldn't do. Click places you aren't meant to click. You'll find bugs and errors that you didn't realize had existed. It'll be difficult to think "out of the box" the first few times, but it gets easier as you keep up the habit AND you eventually end up with a better product.

We will also be implementing unit testing in the coming days. It's still relatively a new subject for us, so check back on these docs for future guidelines.

When you're approaching finalization on certain features/aspects, and you've done all the user testing you think you could possibly do, notify a project manager. We'll have either a designer or another developer review your work from other perspectives to hopefully catch even more hidden issues.

# Quality Assurance

Really just another word for more testing. At this point, the project is mostly done. However, that doesn't mean we're finished! It'll be a few cycles of code reviews, re-building, and more user testing before we even think about live deployment.

# Deployment

My favourite part. Now, during the previous steps, you'll be deploying as well, but those deployments will likely be to a staging server. Now it's time to make it available to the whole world. This step is usually where I'm the most excited, and also the most nervous. But if you followed all the other steps thoroughly, this part will also be the least amount of work.

We have a tradition of ringing a gong when a project goes live, but with the pandemic we no longer have a office to do it in. Maybe we can get Terence to do it during our standup meetings?
